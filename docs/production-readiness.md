# Production Readiness

jobbot3000 production mode is a static, browser-local application tracker. The
container serves HTML, CSS, JavaScript, the web manifest, and health endpoints on
port `8080`; tracker records are stored only in the user's browser IndexedDB.

See [Static tracker observability contract](observability.md) for blackbox smoke checks, resource signals, synthetic staging journeys, and telemetry privacy boundaries.

## Architecture summary

- Public deployment surface: `/`, `/tracker`, `/healthz`, `/livez`, and static
  assets generated by `npm run build`.
- Private state: applications, contacts, outreach messages, notes, interviews,
  offers, artifacts, reminders, settings, and imported backups stay in IndexedDB
  for the current browser profile.
- Release artifacts: image `ghcr.io/futuroptimist/jobbot3000` and chart
  `oci://ghcr.io/futuroptimist/charts/jobbot3000`.
- Kubernetes must receive only app configuration needed to serve static files.

## Safe to deploy publicly

It is safe to publish the static production image/chart because they contain no
real tracker database and expose no write API for private tracker data. The app
shell, CSS, JavaScript, health endpoints, and generic docs are public assets.

## Never store in server-side artifacts

Do not put real job-application data, resumes, emails, Drive links, outreach
messages, contacts, notes, browser exports, SQLite databases, `.env` files, or
local `data/` directories in Docker images, Helm charts, Helm values, ConfigMaps,
Secrets, PVCs, logs, repo fixtures, or public object storage.

## Backup formats

- CSV is the spreadsheet-compatible one-row-per-application format. It is best
  for Google Sheets review and manual editing.
- JSON is the canonical full-fidelity backup bundle for complete restore.
- NDJSON is the line-oriented full-fidelity stream for diffing, review, and
  resilient transfer.

## First-use checklist

1. Open the deployed app in the browser profile you intend to use.
2. Import an anonymized/fake CSV or a current spreadsheet export through the UI.
3. Confirm row counts, conflict warnings, representative companies/roles, and
   follow-up dates.
4. Immediately export JSON and NDJSON backups and store them privately.
5. Export CSV if you still want a spreadsheet-compatible checkpoint.

## First production deploy checklist

1. Publish the image with an immutable tag such as `main-SHORTSHA`.
2. Publish the Helm chart only when chart content changed.
3. Deploy staging with the immutable image tag.
4. Verify `/`, `/tracker`, `/healthz`, and `/livez`.
5. Import a fake or anonymized backup in the staging browser UI.
6. Export JSON/NDJSON from staging and verify record counts.
7. Promote production with the same immutable image tag.

## Restore checklist

1. Use an empty or disposable browser profile when rehearsing a restore.
2. Prefer JSON for canonical restore; use NDJSON when you need line-oriented
   inspection first.
3. Dry-run/preview when available and compare store counts before replacing data.
4. Restore only after saving the current browser's JSON/NDJSON backup.
5. Re-export after restore and compare application counts and representative
   records.

## Rollback checklist

1. Roll the Kubernetes deployment back to the previous immutable image tag.
2. Re-check `/healthz` and `/livez`.
3. Browser IndexedDB data should remain intact across static app rollbacks.
4. If data was cleared or replaced locally, restore the last known-good JSON or
   NDJSON backup through the browser UI.

## Verification checklist

- `npm ci`, `npm run lint`, `npm run typecheck`, `npm run test:ci`, and
  `npm run build` pass.
- Container smoke covers `/`, `/tracker`, `/healthz`, and `/livez`.
- Helm lint/template tests confirm probes, image repository, and no user-data PVC
  or Secret defaults.
- No real data appears in fixtures, chart values, `.env`, Docker context, or
  generated logs.

## Known limitations and follow-ups

- Browser profiles are isolated; dev, staging, and production do not share data
  unless the user imports the same backup into each profile.
- CSV is intentionally lossy for multiple contacts, outreach messages,
  interviews, offers, artifacts, and reminders on one application.
- There is no server-side account sync, auth, SaaS database, or managed backup.
